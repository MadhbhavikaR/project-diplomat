## Agent Development Kit (ADK) Web-Created Agents \& YAML Configuration: Comprehensive Research Report

### Executive Summary

Google's Agent Development Kit (ADK) enables developers and non-technical users to build sophisticated multi-agent AI systems through three complementary methods: Python code, declarative YAML configuration, and a visual builder interface. ADK's YAML-based agent configuration, introduced in version 1.12.0, provides a no-code approach to defining agents, tools, and multi-agent orchestration patterns. This report details the structure of YAML files generated by ADK web interfaces, their configuration schemas, and how they function within the broader ADK ecosystem.[^1][^2][^3]

### Part 1: ADK Web Interface \& Agent Creation Methods

#### Overview of ADK Web

The ADK web interface serves as both a development environment and interactive testing platform for AI agents. Running the command `adk web` launches a local development server (typically on `http://localhost:8000`) that provides three distinct execution modes:[^4][^5]

- **Development UI**: An interactive chat interface with debugging capabilities, displaying agent state, user requests, and message events
- **Terminal Execution**: Command-line execution via `adk run`
- **API Server Mode**: REST API endpoint via `adk api_server`


#### Three Approaches to Agent Creation

ADK supports three distinct methodologies for building agents, each with specific use cases:

**1. Imperative Python Approach**[^4]
The traditional method uses Python code with the ADK Agent class:

```python
from google.adk.agents import Agent

root_agent = Agent(
    model='gemini-2.5-flash',
    name='root_agent',
    description='A helpful assistant',
    instruction='Answer user questions to the best of your knowledge'
)
```

**2. Declarative YAML Configuration**[^3][^1]
The Agent Config feature enables configuration-driven development without code:

```
adk create --type=config my_agent
```

This generates a `my_agent/` folder containing `root_agent.yaml` and `.env` files that can be edited directly.

**3. Visual Builder Interface**[^6][^7]
The experimental Visual Builder provides a graphical, drag-and-drop environment for designing agents. Users can create agents through natural language prompts to an AI assistant, which generates YAML configuration automatically. The tool includes a canvas view showing agent hierarchy, configuration panels, and real-time testing capabilities.[^6]

### Part 2: YAML File Structure \& Schema

#### Project Directory Structure Generated by ADK Web

When creating agents with ADK web, the tool generates a standardized directory structure. For basic configuration-based agents:[^3]

```
my_agent/
├── .env                      # Environment variables and API keys
├── root_agent.yaml           # Primary agent configuration
└── __init__.py               # Python package marker
```

For advanced projects with custom tools and Visual Builder output:[^7]

```
DiceAgent/
├── root_agent.yaml           # Published/production agent version
├── sub_agent_1.yaml          # Sub-agent configurations (if needed)
├── tools/                    # Custom tools directory
│   ├── __init__.py
│   └── dice_tool.py          # Tool implementations
└── tmp/                      # Temporary directory (Visual Builder)
    └── DiceAgent/
        └── root_agent.yaml   # Draft/staging version
```

This hierarchical structure allows agents to reference sub-agents via relative paths, maintaining modularity and enabling multi-agent orchestration.[^8][^7]

#### root_agent.yaml Schema

The root YAML configuration file follows a standardized schema that can be validated against a JSON schema hosted on GitHub. All YAML files should include the schema language server directive for IDE support:[^9][^3]

```yaml
# yaml-language-server: $schema=https://raw.githubusercontent.com/google/adk-python/refs/heads/main/src/google/adk/agents/config_schemas/AgentConfig.json
```

**Core Required and Optional Fields:**[^9][^3]


| Field | Required | Type | Description |
| :-- | :-- | :-- | :-- |
| **agent_class** | Required* | String | Agent type: `LlmAgent`, `SequentialAgent`, `ParallelAgent`, `LoopAgent` (defaults to LlmAgent if empty) |
| **name** | Required | String | Unique identifier for the agent |
| **instruction** | Required (LlmAgent) | String | System prompt that guides agent behavior |
| **model** | Optional | String | LLM model (e.g., "gemini-2.5-flash", "gemini-2.0-flash"); inherited from parent if not specified |
| **description** | Optional | String | Human-readable summary of agent purpose |
| **sub_agents** | Optional | Array | Child agents for multi-agent systems |
| **tools** | Optional | Array | Tools the agent can access |
| **output_key** | Optional | String | Named key for storing agent output in session state |
| **output_schema** | Optional | Object | JSON schema defining expected output structure |
| **input_schema** | Optional | Object | JSON schema defining expected input format |

#### Minimal Agent Configuration Example[^3]

```yaml
# yaml-language-server: $schema=...

name: assistant_agent
model: gemini-2.5-flash
description: A helper agent that can answer users' questions.
instruction: You are an agent to help answer users' various questions.
```


### Part 3: Agent Class Types \& Their YAML Configurations

ADK defines four primary agent classes, each with distinct operational characteristics:

#### 1. LlmAgent - Language Model-Driven Agents[^10][^9]

LlmAgents use Large Language Models as their reasoning core. They support:

- Flexible, language-centric task execution
- Dynamic tool selection and use
- Natural language understanding and generation

**YAML Configuration Example:**[^3]

```yaml
agent_class: LlmAgent
name: search_agent
model: gemini-2.0-flash
description: Performs Google search and answers questions
instruction: You are an agent whose job is to perform Google search queries and answer questions about the results.
tools:
  - name: google_search
```

**LlmAgent-Specific Fields:**[^9]

- `generate_content_config`: Fine-tune LLM generation parameters (temperature, top_k, top_p, max_tokens)
- `before_model_callbacks` / `after_model_callbacks`: Hooks for pre/post LLM processing
- `before_tool_callbacks` / `after_tool_callbacks`: Hooks for tool execution
- `disallow_transfer_to_parent`: Prevent delegating to parent agents
- `disallow_transfer_to_peers`: Prevent delegating to sibling agents


#### 2. SequentialAgent - Deterministic Workflow Orchestration[^11][^10]

SequentialAgents execute sub-agents in a predefined, sequential order without LLM-driven decisions. They support data flow between agents through state passing.

**YAML Configuration Example:**[^12]

```yaml
name: data_pipeline
agent_class: SequentialAgent
sub_agents:
  - config_path: ./validator_agent.yaml
  - config_path: ./processor_agent.yaml
  - config_path: ./reporter_agent.yaml
```

**Data Flow Pattern:**[^12]
Sub-agents coordinate through `output_key` and template substitution:

```yaml
# validator_agent.yaml
name: validator
agent_class: LlmAgent
instruction: Validate the input data.
output_key: validation_status

---
# processor_agent.yaml
name: processor
agent_class: LlmAgent
instruction: Process data if {validation_status} is 'valid'.
output_key: result
```


#### 3. ParallelAgent - Concurrent Task Execution[^10][^11]

ParallelAgents run all sub-agents concurrently, maximizing throughput for independent tasks. Results from all sub-agents are combined.

**Use Case:** Running multiple research agents simultaneously before synthesis.

**YAML Pattern:**

```yaml
name: parallel_research
agent_class: ParallelAgent
sub_agents:
  - config_path: ./renewable_researcher.yaml
  - config_path: ./ev_researcher.yaml
  - config_path: ./carbon_researcher.yaml
```


#### 4. LoopAgent - Iterative Execution[^13][^10]

LoopAgents repeatedly execute their sub-agents for a specified number of iterations, useful for refinement loops or polling workflows.

**YAML Configuration:**

```yaml
name: refinement_loop
agent_class: LoopAgent
max_iterations: 5
sub_agents:
  - config_path: ./generator_agent.yaml
  - config_path: ./reviewer_agent.yaml
```


### Part 4: Tools Configuration \& Implementation

ADK supports multiple tool types within YAML configuration:

#### Built-In Tools[^9]

ADK provides pre-built tools accessible by name:

```yaml
tools:
  - name: google_search          # Google Search integration
  - name: load_memory            # Memory access
  - name: url_context            # Web page content retrieval
  - name: enterprise_web_search  # Enterprise search
  - name: exit_loop              # Loop control
```


#### User-Defined Tools[^9]

User-defined tools can be referenced via fully qualified Python paths:

**Tool Instance Reference:**

```yaml
tools:
  - name: my_package.my_module.my_tool_instance
```

**Tool Class with Arguments:**

```yaml
tools:
  - name: my_package.my_module.MyToolClass
    args:
      param1: value1
      param2: value2
```

**Tool Factory Function:**

```yaml
tools:
  - name: my_package.my_module.create_tool
    args:
      database_url: "postgres://localhost"
      api_key: "${EXTERNAL_API_KEY}"
```

Tool factory functions must conform to the signature:

```python
def create_tool(args: ToolArgsConfig) -> BaseTool:
    ...
```


#### Custom Tool Example[^3]

```yaml
agent_class: LlmAgent
name: prime_agent
description: Checks if numbers are prime.
instruction: |
  You are responsible for checking whether numbers are prime.
  When asked to check primes, you must call the check_prime tool.
  Never attempt to determine prime numbers manually.
tools:
  - name: ma_llm.check_prime
```


### Part 5: Environment Configuration (.env File)

The `.env` file stores sensitive configuration and environment variables required for agent execution.[^3]

#### Configuration for Google Gemini API

```
GOOGLE_GENAI_USE_VERTEXAI=0
GOOGLE_API_KEY=<your-api-key-from-google-ai-studio>
```


#### Configuration for Vertex AI (Google Cloud)

```
GOOGLE_GENAI_USE_VERTEXAI=1
GOOGLE_CLOUD_PROJECT=<your-gcp-project-id>
GOOGLE_CLOUD_LOCATION=us-central1
```

The `.env` file is sourced by ADK at runtime to populate required credentials and configuration parameters.

### Part 6: Multi-Agent System Design Patterns

#### Sub-Agents Pattern[^14]

A root agent intelligently delegates tasks to specialized sub-agents based on task type:

```yaml
# root_agent.yaml
name: root_agent
agent_class: LlmAgent
model: gemini-2.5-flash
instruction: |
  You are a learning assistant.
  Analyze the user question:
  - If it's about programming/coding, delegate to code_tutor_agent
  - If it's about math, delegate to math_tutor_agent
description: Learning assistant that provides tutoring

sub_agents:
  - config_path: ./code_tutor_agent.yaml
  - config_path: ./math_tutor_agent.yaml

---
# code_tutor_agent.yaml
name: code_tutor_agent
agent_class: LlmAgent
model: gemini-2.5-flash
instruction: |
  You are an expert Python tutor.
  Provide clear explanations of programming concepts.
  Generate example code when appropriate.

---
# math_tutor_agent.yaml
name: math_tutor_agent
agent_class: LlmAgent
model: gemini-2.5-flash
instruction: |
  You are an expert mathematics tutor.
  Explain math concepts step-by-step.
  Show worked solutions.
```


#### Sequential Pipeline Pattern[^12]

Three-stage data processing pipeline using state passing:

```yaml
# root - orchestrator (SequentialAgent)
name: data_pipeline
agent_class: SequentialAgent
description: Three-stage data processing pipeline

sub_agents:
  - config_path: ./stage_1_validator.yaml
  - config_path: ./stage_2_processor.yaml
  - config_path: ./stage_3_reporter.yaml

---
# stage_1_validator.yaml
name: validator
agent_class: LlmAgent
instruction: Validate the input data for quality and format compliance.
output_key: validation_result

---
# stage_2_processor.yaml
name: processor
agent_class: LlmAgent
instruction: |
  Process the data validated in the previous stage.
  Access the validation result via: {validation_result}
  Transform and enrich the data.
output_key: processed_data

---
# stage_3_reporter.yaml
name: reporter
agent_class: LlmAgent
instruction: |
  Generate a comprehensive report from the processed data.
  Previous result: {processed_data}
  Format as structured JSON.
```


### Part 7: Advanced YAML Features

#### Structured Input/Output Schemas[^15]

Define strict input and output formats using JSON schemas:

```yaml
agent_class: LlmAgent
name: capital_finder
instruction: Find and return the capital of the requested country.

input_schema:
  type: object
  properties:
    country_name:
      type: string
      description: Name of the country
  required:
    - country_name

output_schema:
  type: object
  properties:
    capital:
      type: string
      description: The capital city
    country:
      type: string
      description: The country name
    population:
      type: integer
      description: Population of capital
  required:
    - capital
    - country
```


#### Model Inheritance[^9]

Child agents inherit the model from parent if not explicitly specified:

```yaml
# root_agent.yaml
name: root_agent
model: gemini-2.5-flash
agent_class: SequentialAgent
sub_agents:
  - config_path: ./sub_agent_1.yaml    # Inherits gemini-2.5-flash
  - config_path: ./sub_agent_2.yaml    # Inherits gemini-2.5-flash

---
# sub_agent_1.yaml (no model specified - inherits from parent)
name: sub_agent_1
agent_class: LlmAgent
instruction: Process the data.

---
# sub_agent_2.yaml (explicit model override)
name: sub_agent_2
model: gemini-2.0-flash                # Override inherited model
agent_class: LlmAgent
instruction: Generate responses.
```


#### Callbacks for Custom Logic[^9]

Pre and post-processing hooks at agent and model levels:

```yaml
agent_class: LlmAgent
name: monitored_agent
instruction: Your instructions here.

before_agent_callbacks:
  - name: my_library.callbacks.log_agent_start

after_agent_callbacks:
  - name: my_library.callbacks.log_agent_result

before_model_callbacks:
  - name: my_library.callbacks.validate_prompt

after_model_callbacks:
  - name: my_library.callbacks.parse_response
```


### Part 8: Visual Builder Output \& Workflow

The ADK Visual Builder generates production-ready YAML files from graphical interactions.[^7]

#### Visual Builder Features[^6]

- **Zero YAML Prototyping**: Build complex workflows without manual YAML editing
- **Visual Agent Mapping**: Interactive node canvas showing agent hierarchy and relationships
- **AI-Powered Assistant**: Generate agents from natural language descriptions
- **Real-Time Testing**: Built-in chat interface for immediate feedback
- **Model Flexibility**: Support for different Gemini models per agent (2.5 Pro for reasoning, Flash for speed)


#### Generated Project Layout[^7]

```
DiceAgent/
├── root_agent.yaml              # Published/production version
├── sub_agent_1.yaml             # Generated sub-agents
├── sub_agent_2.yaml
├── tools/                       # Tools directory
│   ├── __init__.py
│   └── dice_tool.py             # Auto-generated tool code
└── tmp/                         # Draft storage (Visual Builder)
    └── DiceAgent/
        └── root_agent.yaml      # Staging version
```

The Visual Builder maintains both production (`root_agent.yaml`) and draft versions, enabling iterative development without breaking deployed agents.[^16]

### Part 9: Supported Tools \& Limitations

#### Currently Supported Built-In Tools[^3]

ADK Agent Config provides limited but growing support for built-in tools:


| Tool | Status | Notes |
| :-- | :-- | :-- |
| `google_search` | ✓ Supported | Web search integration |
| `load_artifacts` | ✓ Supported | Load external data |
| `url_context` | ✓ Supported | Web page content |
| `exit_loop` | ✓ Supported | Loop control |
| `preload_memory` | ✓ Supported | Memory initialization |
| `get_user_choice` | ✓ Supported | User input |
| `enterprise_web_search` | ✓ Supported | Enterprise search |
| `load_web_page` | ✓ Supported | Requires fully-qualified path |
| `AgentTool` | ✗ Not Supported | Agent as tool |
| `LongRunningFunctionTool` | ✗ Not Supported | Long-duration tasks |
| `VertexAiSearchTool` | ✗ Not Supported | Vertex AI Search |
| `McpToolset` | ✗ Not Supported | MCP integration |
| `ExampleTool` | ✗ Not Supported | Example tools |

#### Agent Type Limitations[^3]

The following agent types are not yet supported in Agent Config format:

- `LangGraphAgent`
- `A2aAgent`


#### Model Support[^3]

- **Supported**: Gemini models only (currently)
- **In Progress**: Third-party model integration
- **Limited**: Cannot use Claude, GPT-4, or other non-Gemini models with YAML config


#### Programming Language Constraints[^3]

- **Tools**: Python only (for agent config)
- **Custom logic**: Python required for callbacks and tool factory functions


### Part 10: ADK CLI Commands \& Execution

#### Core Commands[^5][^4][^3]

```bash
# Create a new config-based agent project
adk create --type=config my_agent

# Launch Visual Builder (web interface)
adk web

# Run agent in terminal
adk run

# Start API server mode
adk api_server

# Deploy to Cloud Run
adk deploy cloud_run

# Deploy to Agent Engine
adk deploy agent_engine --env_file .env
```


#### Running Agents[^5][^4]

Once created, agents can be run through multiple interfaces:

1. **Web UI** (`adk web`): Interactive chat with debugging tools
2. **Terminal** (`adk run`): Direct CLI execution with streaming output
3. **API Server** (`adk api_server`): REST endpoints for external integration
4. **Cloud Deployment**: Via `adk deploy` commands to production environments

### Part 12: Experimental Features \& Considerations

#### Agent Config Experimental Status[^14][^3]

The Agent Config feature is experimental (as of January 2026) with known limitations:

- **Breaking changes possible**: API may evolve significantly
- **Feature gaps**: Not all ADK capabilities available in YAML format
- **Tool support partial**: Limited to supported tools listed above
- **Feedback encouraged**: Google welcomes community input


#### Best Practices When Using YAML Configs

1. **Schema Validation**: Always include the language server schema directive for IDE support
2. **Relative Paths**: Use relative paths for `config_path` references
3. **Environment Variables**: Store sensitive data in `.env`, not in YAML
4. **Modular Structure**: One agent per YAML file for reusability
5. **Clear Instructions**: System prompts (`instruction`) should be explicit and detailed
6. **Output Keys**: Use meaningful `output_key` values for state passing clarity
7. **Version Control**: Store YAML files in git; use `.gitignore` for `.env`

### Conclusion

ADK's YAML-based agent configuration represents a significant advancement in making AI agent development accessible to both technical and non-technical users. The web interface, combined with the Agent Config feature and Visual Builder, enables rapid prototyping of complex multi-agent systems without writing code. The structured YAML schema provides clear configuration pathways while maintaining the flexibility needed for sophisticated orchestration patterns.

The directory structure generated by ADK web interface standardizes project organization, supporting both simple single-agent applications and complex multi-agent hierarchies. While currently experimental with some tool and model limitations, the YAML approach demonstrates Google's commitment to democratizing agent development. Organizations looking to build production-grade agents should follow the documented structure, utilize the provided tools, and reference the official schema for validation.

### Key Takeaways

1. **Three Creation Methods**: Python code, YAML configuration, or visual builder—choose based on use case
2. **Standard Structure**: ADK web generates consistent directory layouts enabling scalability
3. **Four Agent Types**: LlmAgent, SequentialAgent, ParallelAgent, LoopAgent each serve distinct orchestration needs
4. **State Management**: Output keys enable sophisticated data flow in multi-agent pipelines
5. **Experimental Status**: Agent Config feature is production-capable but may evolve with breaking changes
6. **Tool Ecosystem**: Built-in tools available, with custom tool support via Python functions
7. **IDE Support**: YAML schema validation enables autocomplete and real-time error detection
